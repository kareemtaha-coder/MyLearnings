

### **1. The Logical Zone (Expression Tree Construction)**
- **متى تحدث؟** في لحظة الـ **Compilation** (المترجم هو اللي بياخد القرار بناءً على نوع الوعاء `IQueryable`).
- - **ماذا يحدث تقنياً؟**
    
    1. يتم إيقاف تحويل الكود إلى أوامر تنفيذية (IL).
        
    2. يتم تحويل الكود إلى **Object Graph** (هيكل بياني) يتكون من عقد (Nodes).
        
- **النتيجة؟** كود "موصوف" وليس "منفذ"، مما يسمح للـ EF Core بفتحه وقراءته وفهمه.



### Query Compilation

### **المحطة الثانية: الـ Parameter Extraction (استخراج القيم)**

دلوقتي الـ EF Core استلم الـ **Expression Tree** (الخريطة المكشوفة). أول حاجة بيعملها "خلف الكواليس" هي عملية تنظيف وفصل للقيم.

تخيل إنك كاتب الكود ده:

C#

```
int minAge = 20;
var users = context.Users.Where(u => u.Age > minAge).ToList();
```

الـ EF Core بيبص على الشجرة ويلاقي عقدة (Node) نوعها `MemberAccess` بتشاور على المتغير `minAge`. هنا المحرك بيعمل حركة ذكية جداً:

#### **1. الـ How (الميكانيكا):**

المحرك بيمشي على الشجرة (بنمط الـ Visitor) وبيعمل حاجة اسمها **Partial Evaluation**.

- بيلاقي المتغير `minAge` قيمته `20`.
    
- بيقوم "بقص" المتغير ده من الشجرة، وبيحط مكانه **Placeholder** (خانة فاضية) وبيركن القيمة `20` على جنب في قائمة اسمها الـ **Parameters**.

#### **2. الـ Why (السبب التقني):**

ليه ما يترجمش الـ `20` دي جوه نص الـ SQL مباشرة (يعني ليه ما يكتبش `WHERE Age > 20`)؟ لسببين في غاية الأهمية:

- **أ. الـ SQL Injection Protection:** لو القيمة دي جاية من مستخدم (Input)، وفضلنا حاطينها كـ String جوه الـ SQL، ممكن المستخدم يبعت كود خبيث يمسح الداتابيز. لما بنفصلها كـ `Parameter` (@p0)، الداتابيز بتتعامل معاها كـ "قيمة مجردة" مش كـ "كود".
    
- **ب. الـ Query Plan Caching (مهم جداً للـ Performance):** سيرفر قاعدة البيانات (SQL Server) بيعمل Cache لخطة تنفيذ الاستعلام بناءً على "نص الاستعلام".
    
    - لو نص الاستعلام هو: `WHERE Age > 20` وبعد شوية بقى `WHERE Age > 30`.. الداتابيز هتفتكرهم استعلامين مختلفين وتتعب نفسها في بناء خطة جديدة (Execution Plan).
        
    - لكن لو النص ثابت: `WHERE Age > @p0`.. الداتابيز هتعمل Cache للخطة دي وتستخدمها مليون مرة مهما تغيرت قيمة `@p0`.
    
**الخلاصة في المحطة دي:** الـ EF Core "بيفكك" الشجرة، بيفصل **المنطق الثابت** (الشرط) عن **البيانات المتغيرة** (القيم)، عشان يضمن الأمان والسرعة.

هل الخطوة دي واضحة تقنياً (ليه بنفصل القيم عن المنطق)؟ عشان ندخل في المحطة التالتة وهي **الـ Query Caching** (إزاي EF Core بيوفر وقت الترجمة نفسه).


### **Query Caching (Compiled Query Cache)**

#### **1. الـ How (الميكانيكا):**

بعد ما الـ EF Core خلص المحطة اللي فاتت (Parameter Extraction) وفصل القيم عن المنطق، بقى معاه "شكل" أو "هيكل" ثابت للاستعلام (Query Shape/Topology).

1. **توليد الـ Cache Key:** الـ EF Core بياخد الـ Expression Tree (بعد ما شال منها القيم وحط مكانها Parameters) وبيعمل لها **Hashing**. الـ Hash ده هو "البصمة" الفريدة لشكل الاستعلام ده.
    
2. **البحث في الـ Cache:** بيروح لقاموس في الذاكرة (ConcurrentDictionary) ويسأل: "هل البصمة دي موجودة عندي؟".
    
    - **في حالة الـ Cache Hit (موجود):** بيسحب كائن اسمه `CompiledQuery` (وده جواه الـ SQL الجاهز وطريقة تنفيذه) وينط فوراً لمحطة التنفيذ.
        
    - **في حالة الـ Cache Miss (مش موجود):** بيكمل لعملية الترجمة (Translation)، وبعد ما يخلصها، بيخزن النتيجة في الـ Cache عشان المرة الجاية.
        

#### **2. الـ Why (السبب التقني):**

- **توفير دورات المعالج (CPU Cycles):** عملية الـ `ExpressionVisitor` اللي بتمشي على الشجرة وتقرأ الـ Metadata معقدة. الكاش بيخلي العملية دي تحصل **مرة واحدة فقط** لكل شكل استعلام طوال فترة حياة الأبلكيشن.
    
- **السرعة (O(1) vs O(n)):** البحث في الـ Dictionary بالـ Hash بياخد وقت ثابت (Constant Time)، مهما كان الاستعلام معقد، بينما ترجمة استعلام فيه 10 Joins وشروط كتير هتاخد وقت طويل في كل مرة.



### **نقطة تقنية للمحترفين (The Technical Catch):**

ليه الـ **Parameter Extraction** (اللي شرحناها في الخطوة اللي فاتت) كانت ضرورية عشان الكاش ده يشتغل؟

تخيل لو الـ EF Core مافصلش القيم:

- طلب 1: `Where(u => u.Age > 20)` -> الـ Hash هيكون لـ "أكبر من 20".
    
- طلب 2: `Where(u => u.Age > 30)` -> الـ Hash هيكون لـ "أكبر من 30".
    

هنا الـ EF Core هيعتبرهم طلبين مختلفين تماماً وهيعمل ترجمة مرتين! لكن لما فصلنا الـ 20 والـ 30 كـ Parameters، بقى شكل الشجرة في الحالتين هو: `Where(u => u.Age > @p0)`. **النتيجة:** الطلبين ليهم نفس الـ Hash، فالتاني هيتجاب من الكاش فوراً.


**الخلاصة التقنية للنوتس:** الـ Query Caching في EF Core مش بيكاش "البيانات" اللي راجعة من الداتابيز، هو بيكاش **"خطة العمل" (The Translation Result)**. هو بيحول عملية الترجمة من مجهود ذهني شاق للمحرك، لمجرد عملية "نسخ ولصق" لاستعلام تم تجهيزه مسبقاً.



- **ج. الـ Translation (التحويل الفعلي):** هنا بيجي دور الـ **Database Provider** (زي SQL Server Provider). بيمشي على الشجرة باستخدام نمط الـ **Visitor Pattern**، ويبدأ يترجم كل عقدة لما يقابلها في SQL.
    
    - `Where` تتحول لـ `WHERE`.
        
    - `Any` تتحول لـ `EXISTS`.
        
    - `Skip/Take` تتحول لـ `OFFSET/FETCH`.

### 3. مرحلة الـ SQL Generation (توليد الكود النهائي)

بعد ما الـ Provider يخلص ترجمة الشجرة، بيطلع مخرج نهائي عبارة عن **SQL String** ومعاه مجموعة من الـ **Parameters**. في المرحلة دي، الـ SQL بيكون "Optmized" على قد ما يقدر (مثلاً بيختار الأسماء المستعارة للجداول `t0`, `t1` وهكذا).

---

### 4. مرحلة الـ Execution (الوصول للداتابيز)

هنا بنخرج من عباءة EF Core وندخل في عباءة **ADO.NET** (الأساس اللي مبني عليه كل ده):

1. **Open Connection:** الـ EF Core بيطلب من الـ `DbConnection` إنه يفتح اتصال بقاعدة البيانات (أو ياخد واحد جاهز من الـ Connection Pool).
    
2. **Command Creation:** بيعمل كائن `DbCommand` ويحط فيه نص الـ SQL والـ Parameters.
    
3. **Round-trip:** بيتم إرسال الطلب عبر الشبكة (Network) إلى محرك قاعدة البيانات (SQL Server مثلاً).
    
4. **SQL Execution:** الداتابيز بتنفذ الاستعلام، بتعمل Query Plan، وبتطلع النتائج.
    
5. **Streaming Results:** الداتابيز مش بتبعت الداتا كلها مرة واحدة في طرد بريد، هي بتفتح **Cursor** (مؤشر) وتبدأ تبعت الداتا على شكل **Streams of Rows** (تدفق من الصفوف).
    

---

### 5. محطة الـ DbDataReader (الاستقبال)

الـ EF Core بيستقبل أول "صف" (Row) من الداتابيز عن طريق كائن اسمه `DbDataReader`.

- في اللحظة دي، الداتا لسه عبارة عن قيم أولية (Integers, Strings, Dates) جوه الـ Reader.
    
- لم يتم إنشاء أي C# Objects بعد.
