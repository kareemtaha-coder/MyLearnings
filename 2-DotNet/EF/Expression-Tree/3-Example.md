تثبيت المفاهيم بـ **Example** واحد هو أفضل وسيلة لهضم هذه الرحلة التقنية. سنأخذ استعلاماً بسيطاً ونمرره عبر كل هذه "البوابات" الهندسية.

**المثال الموحد:** جلب المستخدمين النشطين الذين لديهم رصيد أكبر من مبلغ معين.

C#

```
decimal minBalance = 500;
var users = context.Users
    .Where(u => u.IsActive && u.Balance > minBalance)
    .ToList();
```

---

### المرحلة الأولى: The Logical Zone (Expression Tree)

- **ماذا يحدث؟** المترجم (Roslyn) يرى `IQueryable` فيقوم بـ "تجميد" المنطق.
    
- **التمثيل داخل الذاكرة (Object Graph):** يتم بناء شجرة كائنات:
    
    - `MethodCallExpression`: تنادي دالة `Where`.
        
    - `LambdaExpression`: تمثل المدخل `u`.
        
    - `BinaryExpression`: نوعها `AndAlso` (للربط بين الشرطين).
        
    - **الفرع الأول:** `MemberAccess` للخاصية `IsActive`.
        
    - **الفرع الثاني:** `BinaryExpression` نوعها `GreaterThan`.
        
        - يسار: `MemberAccess` للخاصية `Balance`.
            
        - يمين: `ConstantExpression` يشير للمتغير الخارجي `minBalance`.
            

---

### المرحلة الثانية: Query Compilation (الاستخراج والبحث)

#### أ. الـ Parameter Extraction:

- **العملية:** المحرك يمر على الشجرة، يجد `minBalance`.
    
- **النتيجة:** يتم استبدال القيمة `500` بـ `Placeholder` (خانة) تسمى `@p0`. المنطق الآن أصبح: `IsActive == true AND Balance > @p0`. تم فصل الـ **Logic** عن الـ **Data**.
    

#### ب. الـ Query Caching:

- **العملية:** يتم أخذ "شكل" الشجرة (المنطق بدون الـ 500) وعمل `Hash` له.
    
- **النتيجة:** إذا كان هذا أول طلب بهذا الشكل، سيتم تسجيل الـ Hash في الـ `ConcurrentDictionary`. لو كررت الطلب بـ `minBalance = 1000` بعد ثانية، الـ Hash سيكون متطابقاً، فيسحب الـ SQL المترجم فوراً دون إعادة المراحل التالية.
    

#### ج. الـ Translation (التحويل الفعلي):

- **العملية:** الـ SQL Provider يستخدم `Visitor Pattern` لترجمة العقد:
    
    - `Where` -> `SELECT ... FROM [Users] WHERE`.
        
    - `AndAlso` -> `AND`.
        
    - `GreaterThan` -> `>`.
        
- **النتيجة:** منطق C# تحول لقاموس SQL.
    

---

### المرحلة الثالثة: SQL Generation (الصياغة النهائية)

- **العملية:** تجميع الخيوط وكتابة النص النهائي بأسلوب الـ SQL المنشود.
    
- **المخرج التقني:**
    
    SQL
    
    ```
    SELECT [u].[Id], [u].[Name], [u].[Balance], [u].[IsActive]
    FROM [Users] AS [u]
    WHERE [u].[IsActive] = CAST(1 AS bit) AND [u].[Balance] > @p0
    ```
    
- **Optimization:** لاحظ تحويل `bool` إلى `bit` وإضافة `AS [u]` لضمان عدم تضارب الأسماء.
    

---

### المرحلة الرابعة: Execution (ADO.NET)

- **العملية:** الـ EF Core يطلب اتصالاً من الـ Pool ويفتح القناة.
    
- **الخطوات:**
    
    1. إنشاء `DbCommand`.
        
    2. تمرير قيمة `@p0 = 500`.
        
    3. إرسال الأوامر عبر الشبكة للسيرفر.
        
    4. سيرفر الـ SQL يستلم النص، يجد الخطة المناسبة (Execution Plan)، وينفذ البحث على الهارد ديسك/الرامات.
        

---

### المرحلة الخامسة: DbDataReader (الاستقبال الخام)

- **الحالة:** السيرفر بدأ يرسل النتائج.
    
- **الميكانيكا:** الـ `DbDataReader` يفتح "تيار" (Stream).
    
    - الصف الأول: `[1, "Ali", 700, 1]`
        
    - الصف الثاني: `[5, "Sara", 1200, 1]`
        
- **الوضع التقني:** البيانات الآن موجودة في الـ **Application Buffer** (الرامات) لكنها مجرد **Bytes** ونصوص. لا توجد `User Objects` حتى هذه اللحظة.
    

---

**هنا تنتهي رحلة الـ Query Pipeline وتبدأ رحلة الـ Materialization.**

**لماذا توقفنا هنا؟** لأن الخطوة القادمة هي التي ستحول هذه المصفوفات (`[1, "Ali", 700, 1]`) إلى كائن: `new User { Id = 1, Name = "Ali", Balance = 700 ... }`

**هل هذا المثال الموحد جعل الخطوات مترابطة تقنياً في ذهنك؟** إذا كان نعم، يمكننا البدء في شرح **The Materialization Engine**، وكيف يقوم بخلق الكائنات بسرعة خرافية.


### 2. مين بقى اللي بيفشل؟ (المثال الحقيقي للفشل)

الفشل بيحصل لو أنت "حقنت" `IEnumerable` في **نص** الرحلة قبل ما الـ SQL يتكون.

**شوف المثال ده (ده اللي بيفشل في الترجمة):**

C#

```
var query = context.Users
    .AsEnumerable() // <--- هنا الكارثة! أنت حولت المسار لـ IEnumerable بدري
    .Where(u => u.Age > 20) 
    .ToList();
```

**ليه ده بيفشل (تقنياً)؟**

1. بمجرد ما ناديت `.AsEnumerable()`، أنت قلت للـ EF Core: "شكراً لحد كدة، أنا هكمل الباقي كـ Objects عادية".
    
2. المترجم (Roslyn) لما ييجي عند الـ `.Where` اللي بعدها، هيلاقي الوعاء بقى `IEnumerable` فمش هيعمل **Expression Tree**، بل هيعمل **IL Instructions** (صندوق أسود).
    
3. الـ EF Core مش هيشوف الـ `Age > 20` لأنها بقت كود تنفيذي مقفول.
    
4. **النتيجة:** الـ EF Core هيبعت للداتابيز `SELECT * FROM Users` ويجيب مليون سجل يفلترهم في الرامات عندك.
    

---

### 3. مثال الـ Value Object اللي بيفشل (حتى وأنت IQueryable)

فيه نوع تاني من الفشل بيحصل وأنت لسه `IQueryable` (يعني لسه محتفظ بالشجرة)، بس الشجرة جواها "عقدة" الـ SQL مبيفهمهاش.

C#

```
// مثال الـ Value Object
var users = context.Users
    .Where(u => u.Address.IsValid()) // <--- IsValid() دالة C# داخل الـ Value Object
    .ToList();
```

**تحليل الفشل هنا:**

1. أنت لسه `IQueryable` والمترجم فعلاً عمل **Expression Tree**.
    
2. الـ EF Core استلم الشجرة وبدأ يترجمها (محطة الـ Translation).
    
3. وصل عند عقدة اسمها `IsValid()`. سأل القاموس بتاعه: "يا SQL Server، عندك ترجمة لـ `IsValid`؟".
    
4. الـ SQL رد: "لا، معرفش دي إيه".
    
5. **النتيجة:** هنا بيحصل الـ **Translation Exception**. الـ EF Core بيرمي إيرور ويقولك: "أنا شايف طلبك بس مش عارف أترجم الحتة دي لـ SQL".
    

### الملخص للنوتس (الفرق بين الـ Triggers والـ Failures):

| **الفعل**                | **التأثير التقني**          | **النتيجة**                                                                    |
| ------------------------ | --------------------------- | ------------------------------------------------------------------------------ |
| **`.ToList()`**          | محرك تنفيذ (Trigger)        | ينهي الترجمة ويبدأ جلب البيانات بنجاح.                                         |
| **`.AsEnumerable()`**    | تغيير نوع الوعاء في المنتصف | يقطع شجرة التعبيرات ويجبر الـ EF على جلب الداتا كلها (Client-side evaluation). |
| **دالة خارجية (Method)** | عقدة غير قابلة للترجمة      | الـ EF Core بيفشل في تحويل الشجرة لـ SQL ويضرب Exception.                      |





---

### 1. الـ `Func<T, TResult>` (الـ Delegate/المفوض)

الـ `Func` هو عبارة عن **Delegate**. في الـ Memory، هو عبارة عن **Pointer** (مؤشر) بيشاور على "بلوك" من كود الـ IL (لغة الآلة) القابل للتنفيذ.

- **طبيعة الكود (Execution):** بمجرد عمل Compile، الكود اللي جوه الـ `Func` بيتحول لأوامر نهائية للمعالج.
    
- **الرؤية (Black Box):** بالنسبة لأي برنامج تاني (زي EF Core)، الـ `Func` هو صندوق أسود. البرنامج يقدر "ينادي" عليه (Invoke) ويسأله عن النتيجة، لكن مايقدرش "يفتحه" ويشوف هو مكتوب جواه `Age > 20` ولا `Name == "Ali"`.
    
- **مكان العمل:** الرامات (In-Memory) والـ CPU الخاص بجهازك.
    

---

### 2. الـ `Expression<Func<T, TResult>>` (شجرة التعبيرات)

الـ `Expression` هو **Data Structure** (هيكل بيانات). هو مش كود شغال، هو "وصف" للكود.

- **طبيعة الكود (Metadata):** المترجم بيحول الكود بتاعك لكائنات (Objects) بتوصف أجزاء الكود. (ده اللي شرحناه في الـ Object Graph).
    
- **الرؤية (Transparent):** بما إنه عبارة عن كائنات (Nodes)، أي برنامج يقدر "يمشي" جواه ويحلله حتة حتة. يقدر يعرف إن فيه عملية مقارنة، وإيه اسم الـ Property، وإيه القيمة الثابتة.
    
- **مكان العمل:** بيستخدم كـ "خريطة" بتتبعت لمحركات تانية (زي الـ SQL Provider) عشان تترجمها للغة تانية خالص.
    

---

### مقارنة تقنية بالسيناريوهات (Technical Scenarios)

|وجه المقارنة|`Func<User, bool>`|`Expression<Func<User, bool>>`|
|---|---|---|
|**التوصيف التقني**|كود تنفيذي (Opaque Instructions)|كود وصفي (Transparent Metadata)|
|**كيف يراه الـ Compiler**|`Method` (دالة خفية)|`Tree` (شجرة كائنات)|
|**قابلية التحليل**|مستحيل (إلا بـ Reflection معقد جداً للـ IL)|سهل جداً عبر الـ `ExpressionVisitor`|
|**الاستخدام في LINQ**|يستخدم مع `LINQ to Objects` (IEnumerable)|يستخدم مع `LINQ to SQL/Entities` (IQueryable)|
|**التنفيذ**|`delegate.Invoke(user)`|لازم يحصل له `.Compile()` الأول عشان يتحول لـ Func ويشتغل|

---

### مثال يوضح "ليه بنحتاج الـ Expression":

تخيل لو الـ EF Core بيستلم `Func`:

C#

```
Func<User, bool> myFunc = u => u.Age > 20;
// EF Core هنا معاه "عنوان كود" بس. 
// هل يقدر يكتب SQL؟ لا، لأنه مش عارف العنوان ده جواه "Age" ولا "Salary".
```

لكن لو استلم `Expression`:

C#

```
Expression<Func<User, bool>> myExpr = u => u.Age > 20;
// EF Core بيمسك myExpr ويقرأ:
// Body -> BinaryExpression (GreaterThan)
// Left -> MemberAccess (Age)
// Right -> Constant (20)
// "أها! كدة أقدر أكتب: WHERE Age > 20"
```

---

### الخلاصة للنوتس:

- **الـ `Func`:** هو "الماكينة" اللي بتمشي وتطلع إنتاج (كود شغال).
    
- **الـ `Expression`:** هو "الرسم الهندسي" بتاع الماكينة (بيانات بتوصف الماكينة).
    

الـ EF Core محتاج **الرسم الهندسي** عشان يبني ماكينة زيه في الـ SQL، لكنه مابيقدرش يعمل حاجة بـ **الماكينة المقفولة** اللي جاية من الـ C#.
